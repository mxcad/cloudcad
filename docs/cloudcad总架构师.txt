## CloudCAD 项目完整交接文档

### 一、项目背景与起点

**项目定位**：在线 CAD 协同平台，前端基于 mxcad/mxcad-app，后端 NestJS。项目一开始全靠 AI 生成，功能勉强跑通但架构严重腐化。

**初始状态**：
- 后端 75 个 Service，仅 1 个 spec 文件，零测试
- 两个"上帝模块"：MxCad（21 个 Service）和 FileSystem（14 个 Service）
- 5 对模块间循环依赖（均使用 forwardRef）
- 185 个接口无版本号，废弃接口不敢删除
- 前端 React 19 + Zustand + Tailwind，代码耦合严重

**核心痛点**：
- 加功能困难，改一处怕炸一片
- Bug 反复出现，改不完
- 不知道哪些接口在用、哪些废弃
- 前端出身，后端架构概念不熟

**我们达成的共识**：先诊断，再重建。三步走战略——修Bug定终稿 → 全面重构后端 → 重构前端为 Vue 3。

---

### 二、架构设计核心决策

**2.1 总体架构原则**

1. **核心封死，零件可换**：转换引擎是唯一核心壁垒，封装成独立包。其他模块（存储、版本管理、上传）通过接口定义，可替换但不强制。
2. **不做客户扩展区**：用户改样板间是他们的事，升级覆盖了不负责。因此不需要 `extensions/`、`module-registry.ts`、`extension-registry.ts` 等扩展机制。
3. **三层交付模型**：开箱即用（Docker 镜像）、轻量定制（配置文件）、深度定制（带着核心包自建项目）。
4. **Monorepo 骨架**：`apps/`（应用）、`packages/`（库）、`runtime/`（离线运行时依赖）三层分离。

**2.2 关键架构决策记录**

| 决策 | 结论 | 时间 |
|------|------|------|
| 前端技术栈 | 从 React 迁移到 Vue 3 + Vuetify 3，复用 mxcad-app 内部已有的 Vuetify 实例 | 冲刺二完成后 |
| 存储层 | 使用 flydrive 作为文件存取抽象层，通过 IStorageProvider 接口隔离 | 冲刺二期间 |
| 上传模块 | 替换为 @tus/server + Uppy 开源方案 | 冲刺三期间 |
| 版本管理抽象 | 定义 IVersionControl 接口，当前 SvnVersionControl 实现，未来可替换为 Git | 冲刺三后 |
| 认证抽象 | 定义 IAuthProvider 接口，为未来对接官网会员体系预留插口 | 冲刺三后 |
| 权限抽象 | 定义 IPermissionStore 接口，权限数据来源可替换 | 冲刺三后 |
| 图纸库/图块库合并 | 统一为 IPublicLibraryProvider 接口，消除重复代码 | 冲刺三后 |

**2.3 文件管理核心架构**

文件管理是整个平台的心脏。冲刺二将原来的上帝模块拆解为四个层次：

- **业务封装层**：图纸库、图块库、我的图纸、项目管理——本质上都是文件系统的不同视图
- **核心能力层**：8 个独立子模块（FileOperationsService、FileTreeService、FileDownloadExportService、ProjectMemberService、SearchService、FileSystemPermissionService、FileValidationService、FileHashService）
- **基础设施层**：IStorageProvider、IVersionControl、IUploadProvider、IPermissionStore
- **物理存储层**：filesData/、SVN 仓库、uploads/、temp/

**关键存储规则**：
- `uploads/` 是静态素材库，存放以哈希命名的原始文件和转换后的 mxweb
- `data/files/NodeId/` 是动态版本区，存放用户修改后的最新版本和分片文件
- SVN 只提交分片文件（bin），mxweb 不进入版本库
- 文件访问通过两级查找：先找 NodeId 目录，再找 uploads 目录
- 秒传基于哈希，复制区分"未修改"和"已修改"

**2.4 认证与鉴权体系**

- 系统级双认证：Header（Authorization: Bearer）+ Cookie（auth_token）
- mxcad.sid 是旧 Session 标识符，已标记废弃，待逐步替换
- JWT 策略已支持从 Cookie 和 Header 双重提取 Token
- 权限体系分系统权限和项目权限两层，6 种权限检查机制已收敛
- 认证（AuthFacadeService）和授权（PermissionService）在代码层面已分离

## 十一、架构师的思考脉络

### 11.1 我们如何处理“抽象与过度设计”的平衡

在整个重构过程中，我们反复面临一个核心矛盾：**什么该抽象，什么不该抽象。**

**初期倾向**：过度设计。我们一度设计了完整的客户扩展体系——`extensions/` 目录、`module-registry.ts`、`extension-registry.ts`、`IAuthProvider`、`IPermissionStore`、`IStorageProvider` 等全系列接口。这是因为我们的初始假设是“客户需要安全地替换模块”。

**中期转折**：用户明确提出，客户修改样板间是他们自己的事，升级时正常覆盖，不需要为“客户扩展”预留机制。这个决策从根本上改变了架构方向。我们废弃了扩展区的设计，不再为“外部可替换性”进行抽象。

**最终策略**：抽象只为“我们自己的未来”服务。认证抽象是为了未来对接官网会员体系，权限抽象是为了未来可能的多数据源场景，版本管理抽象是为了未来可能从 SVN 切换到 Git。如果未来没有明确的变化需求，就不做抽象。

**核心教训**：抽象的目的不是为了“方便别人”，而是为了“保护我们自己”。我们只需要为“预见的未来”预留插口，不需要为“所有可能的未来”进行过度设计。

---

### 11.2 我们在“前后端优先级”上的摇摆

**初期执念**：尽快进入前端 Vue 3 重构，因为后端“看起来差不多了”。
**现实打脸**：每次深入一个模块，都能发现新问题。版本管理没抽象、权限数据直接耦合 Prisma、文件管理子模块还有绕过存储抽象的 BUG。
**最终定下的铁律**：**后端必须等到所有模块都审计完毕、所有 P0 问题都修复后，才能正式启动冲刺四。** 不能因为“想早点进入前端”而让后端留下已知风险。

---

### 11.3 我们对 AI 能力边界的认识进化

**初期**：凭网上评测和主观印象分配任务。
**中期**：通过实际产出评估每个 AI 的能力。
**最终判断依据**：
- Trea 的实际产出最多、质量最稳定，不仅擅长并发分析，还成功完成了版本管理抽象、公共资源库合并等架构实施任务
- Claude Code 适合做深度分析和最复杂的架构决策，但不适合批量执行
- Mimo 通过热身任务展现了超出预期的综合能力，被提升为冲刺四核心战力
- OpenCode 的边界很清晰——单步执行、简单体力活稳定，复杂任务会卡住

## 十二、重要的技术讨论与结论

### 12.1 就“离线部署”的确认

项目需要支持完全离线部署，所有依赖必须通过 pnpm 的离线缓存机制安装。我们确认了 `@tus/server` 和 `Uppy` 都是纯 JavaScript 库，不依赖任何外部网络服务，完全兼容离线交付模型。`runtime/` 目录是离线运行时依赖的存放位置，不是 `vendor/`（我之前在文档中写错过这个名字）。

### 12.2 就“文件存储物理结构”的确认

- `uploads/` 是静态素材库，存放以哈希命名的原始文件和转换后的 mxweb。这个目录是共享的、只读的。
- `data/files/NodeId/` 是动态版本区，存放用户修改后的最新版本和分片文件。
- SVN 提交的是分片文件（bin），mxweb 不进入版本库。
- 文件访问通过两级查找：先找 NodeId 目录的 `latest.mxweb`，没有就去 `uploads/` 读源文件。
- 历史版本的 mxweb 文件按需生成并缓存，命名规则为 `{basename}_v{revision}.mxweb`。

### 12.3 就“认证与图片加载冲突”的解决

前端通过 `<img>` 标签加载缩略图时，浏览器不会发送 `Authorization` 请求头（只发送 Cookie）。这导致我们在冲刺三中为缩略图端点添加的 `JwtAuthGuard` 认证失败。

**解决方案**：不是创建新的 MixedAuthGuard，而是让系统默认支持双重认证。我们确认了后端在登录时已经设置了 `auth_token` Cookie，JWT 策略已经通过 `ExtractJwt.fromExtractors` 同时支持从 Header 和 Cookie 中提取 Token。这个功能其实已经存在，不需要额外开发。

### 12.4 就“mxcad.sid 的废弃标记”

`mxcad.sid` 是旧的 Session 标识符，只用于服务端状态管理，不参与前端认证。我们已在 `AGENTS.md` 中添加了废弃声明，防止任何 AI 在未来误用这个标识符。它的存在是历史原因，彻底移除需要逐步替换所有依赖 `req.session` 的代码。

## 十三、重要的“被放弃”方案

### 13.1 客户扩展体系

我们曾计划设计完整的扩展机制——`extensions/` 目录物理隔离、`module-registry.ts` 注册后端模块、`extension-registry.ts` 注册前端组件。但最终确认客户改样板间是他们的自由，我们不负责升级覆盖的兼容性。因此整个扩展体系被废弃。

### 13.2 前端 SDK 的接口抽象

我们曾计划在前端 `@cloudcad/sdk` 中定义可替换的服务接口（类似后端的 Provider 模式），但最终确认前端不需要这种机制。前端只需要一个封装好的 API 调用层和全局状态管理，不需要为“可替换性”预留接口。

### 13.3 latest.mxweb 的命名方案

我们曾讨论用固定的 `latest.mxweb` 文件名来指向最新版本，但这个方案因为浏览器缓存问题被否定。实际采用的是 `{nodeId}.{ext}.mxweb` 格式，配合 `?t=更新时间` 参数做缓存破坏。

---

## 十四、各模块的特殊规则

### 14.1 图纸库/图块库的独特逻辑

- 无所有者（ownerId 为 null）
- 公开只读（浏览不需要登录）
- 管理员通过系统权限（LIBRARY_DRAWING_MANAGE / LIBRARY_BLOCK_MANAGE）维护
- 保存时跳过 SVN 版本管理，所有修改原地覆盖
- 删除直接永久删除，不走回收站

### 14.2 我的图纸（个人空间）的本质

通过 `personalSpaceKey` 字段标记，用户创建时自动生成，本质是一个只有自己可见的默认项目。它复用完整的文件系统能力，但限制了访问权限只对所有者可见。

### 14.3 搜索模块的性能隐患

搜索模块的 `getAllProjectNodeIds` 存在 N+1 查询问题——递归逐层查数据库获取子节点。建议后续用 PostgreSQL 递归 CTE 一次性获取所有节点 ID。资源库搜索之前无权限检查（已修复），需要 `LIBRARY_*_MANAGE` 系统权限。

### 14.4 转换引擎的并发控制

转换程序调用本地二进制文件，通过信号量控制并发数（默认最大 3）。这是因为转换非常消耗内存，并发过多会导致服务崩溃。转换引擎的 `conversion-engine` 包在冲刺一中已搭建好骨架，但和实际调用还有差异，需要后续统一。

## 十五、前端技术细节

### 15.1 两个 Vuetify 3 实例共存

mxcad-app 内部有一个 Vuetify 3 实例，外围平台创建另一个独立的实例。两者不共享状态，但通过同一套 JSON 配置文件共享主题。这样 mxcad-app 内部样式永远不会被外围平台污染，外围平台也可以自由升级 Vuetify 版本而不影响 mxcad-app。

### 15.2 样式隔离机制

mxcad-app 内部使用 CSS `:not()` 选择器排除外围平台的样式影响，同时使用 `visibility` 和 `zIndex` 做视图切换。外围平台的 Vuetify 组件不需要重复引入 CSS 文件，直接复用 mxcad-app 已注入全局的 CSS 变量。

### 15.3 前端不需要抽象

我们确认前端不需要像后端那样抽象可替换接口。前端组件的替换成本很低——复制一个文件，改几行引用就完成了。不需要依赖注入那样的重机制。

## 十六、部署运维相关

### 16.1 离线部署体系

项目支持完全离线部署，通过 pnpm 的全局 store 功能将所有依赖打包到 `vendor/packages/` 目录（实际目录名是 `runtime/`），客户在内网环境下执行 `pnpm install --offline` 即可安装所有依赖。这包括 @tus/server、Uppy、flydrive 等所有第三方库。

### 16.2 数据库迁移的铁律

任何对 `schema.prisma` 的修改，必须同时生成迁移文件。迁移文件需要确保幂等性——使用 `IF NOT EXISTS` 等保护关键字。客户升级时通过 `prisma migrate deploy` 自动执行迁移，不能要求手动操作数据库。

### 16.3 打包脚本的路径更新

冲刺一的 Monorepo 骨架迁移改变了目录结构（`packages/backend` → `apps/backend`），打包脚本中的路径需要同步更新。这些脚本还在 `runtime/scripts/` 目录下。

## 十七、我们协作中的经验教训

### 17.1 每次认为“差不多了”时都要冷静

我们在版本管理、权限、文件管理子模块等多次案例中，都经历了“以为没问题了”到“发现新问题”的过程。这说明系统性的架构审计是必须的，不能凭感觉判断“是否完成”。

### 17.2 每次修改前先审计，后决策

我们确立了铁律：不要在不确定的情况下修改代码。先派 AI 做只读审计，搞清楚代码的实际状态，然后再决定是否需要调整方案。这个流程避免了多次基于假设的修改。

### 17.3 AI 派发任务的纪律

- Trea 的无限并发能力非常适合批量审计任务，但不需要拆成多个独立指令，可以一次发给它并发处理
- OpenCode 只能单步执行，不能同时给它多个任务
- 给 Claude Code 派任务时要用精确指令，适应它深度思考的特点
- 每次指令必须明确指定参考文档路径，不能模糊引用“某份报告”

### 17.4 “先对齐后端，后启动前端”的铁律

这个铁律是在经历了多次“急于启动冲刺四”的冲动后被确立的。现在它是不可动摇的原则——后端所有模块必须全部审计完毕、所有 P0 问题必须全部修复，才能正式启动 Vue 3 重构。

## 二十、具体的技术约定

### 20.1 文件存储的命名规范

| 文件类型 | 命名格式 | 示例 | 存储位置 |
|---------|---------|------|---------|
| 原始DWG文件 | `{nodeId}.{ext}` | `node_abc123.dwg` | `data/filesData/YYYYMM/nodeId/` |
| 转换后的MXWeb文件 | `{nodeId}.{ext}.mxweb` | `node_abc123.dwg.mxweb` | 同上 |
| 缩略图 | `thumbnail.{webp/jpg/png}` | `thumbnail.webp` | 同上 |
| 历史版本缓存 | `{basename}_v{revision}.mxweb` | `node_abc123.dwg_v78.mxweb` | 同上 |
| 初始版本备份 | `{basename}_initial.mxweb` | `node_abc123.dwg_initial.mxweb` | 同上 |
| 分片文件(临时) | `{chunkIndex}_{hash}` | `0_abc123def456` | `data/temp/chunk_{hash}/` |
| 上传临时文件 | `{hash}.{ext}.mxweb` | `abc123def.dwg.mxweb` | `uploads/` |

### 20.2 目录分配规则

目录由 `DirectoryAllocator` 服务负责分配，采用年月分级目录结构：
- 主目录格式：`YYYYMM`（如 `202604`）
- 子目录格式：`YYYYMM_N`（N从1开始递增，最多100个子目录）
- 节点目录格式：`node_{timestamp}_{random}`
- 每个年月目录最多容纳300000个节点(`FILES_NODE_LIMIT`)

### 20.3 MD5哈希的使用场景

- 秒传检测：前端计算文件MD5，后端查询 `fileHash` 字段判断文件是否已存在
- 文件去重：不同用户上传相同文件时，通过哈希判断是否已存在
- 哈希命名：临时上传文件以哈希命名，避免文件名冲突
- `FILE_HASH_LENGTH` 常量定义了哈希长度为32

### 20.4 软删除的实现约定

- `deletedAt`：不为null表示已删除，null表示正常
- `deletedByCascade`：标记是否为级联删除（父节点删除导致子节点连带删除）
- `deletedFromStorage`：标记物理文件是否已从存储中删除
- `fileStatus`：文件状态枚举，包含 `DELETED` 状态
- `projectStatus`：项目状态枚举，包含 `DELETED` 状态
- 定时清理：每天凌晨4点自动清理超过30天的软删除记录

## 二十一、安全措施汇总

### 21.1 路径遍历防护

- 文件名验证：禁止 `..` 等路径遍历字符
- 路径安全检查：`ValidationHelper.isValidNodePath()` 函数
- 目录路径验证：`nodeDirectoryPath.endsWith(nodeId)` 检查

### 21.2 文件类型验证

- DWG魔数验证：检查文件头6字节（`AC1.50` - `AC1032`）
- DXF结构验证：检查 `SECTION/HEADER/ENDSEC` 标记
- PDF魔数验证：检查 `%PDF` 文件头
- 扩展名白名单：支持7种字体格式、多种CAD和图片格式

### 21.3 文件名清理

- `sanitizeFilename()` 函数执行7步清洗管道：
  1. 移除路径分隔符
  2. 移除非法字符
  3. 限制长度
  4. 保留扩展名
  5. 去除首尾空白
  6. 处理纯扩展名情况
  7. 缓冲区长度截断

### 21.4 权限检查机制

- 双保险策略：Controller层Guard(门卫) + Service层强制逻辑(保险箱)
- 系统权限：通过 `@RequirePermissions` 装饰器声明
- 项目权限：通过 `@RequireProjectPermission` 装饰器声明
- JWT双认证：同时支持Header和Cookie两种Token来源

## 二十二、性能相关的发现与建议

### 22.1 搜索模块N+1问题

- `getAllProjectNodeIds` 递归逐层查询数据库
- 深层文件树会导致大量数据库往返
- 建议使用PostgreSQL递归CTE一次性获取所有节点

### 22.2 文本搜索无索引

- 当前使用 `LIKE %keyword%` 模式匹配
- 无法使用数据库索引
- 建议实现PostgreSQL全文搜索(GIN/GIST索引)

### 22.3 建议的复合索引

```sql
-- 项目文件搜索
CREATE INDEX IF NOT EXISTS idx_project_search 
ON file_system_nodes (projectId, deletedAt, isFolder, updatedAt);

-- 资源库搜索
CREATE INDEX IF NOT EXISTS idx_library_search 
ON file_system_nodes (libraryKey, deletedAt, isFolder, updatedAt);
```

### 22.4 缓存架构的防护机制

- 缓存穿透保护：`nullTTL: 60s, bloomSize: 1000000`
- 缓存雪崩保护：`randomizationRange: 300s`（TTL随机化）
- 缓存版本控制：默认禁用（`versionConfig.enabled: false`）

### 22.5 字体库的串行读取

- `getFonts()` 方法使用 `for...of` 循环串行读取文件状态
- 建议使用 `Promise.all` 并行处理

## 二十九、关键架构决策记录

### 29.1 为什么不用彻底的SSO
- SSO的“统一入口”是单点故障风险
- 客户各自独立部署，不适合统一认证中心
- 内置用户系统保持产品的独立性和交付能力
- 未来通过接口抽象可平滑对接外部IAM平台

### 29.2 为什么不做扩展区
- 客户修改样板间是他们的自由
- 升级时正常覆盖，不负责兼容性
- 扩展机制会增加架构复杂度和维护成本

### 29.3 为什么前端不做抽象
- 后端有依赖注入，可以做运行时替换
- 前端是静态编译，替换组件只需复制文件
- 前端组件的替换成本很低，不需要接口那样的重机制

### 29.4 为什么悲观锁不够
- 悲观锁会阻塞其他用户读取，影响协作流畅性
- 乐观锁只在保存时检查冲突，大部分时间不阻塞
- CAD协同场景下冲突概率极低

### 29.5 乐观锁的实现决策
- 由CodeBuddy Code自主实现，覆盖前后端
- 前端在文件打开时记录 `updatedAt`
- 保存时通过 FormData 传给后端
- 后端保存前检查 `updatedAt`，不匹配则返回409
- 保存成功后更新前端缓存的 `updatedAt`

## 三十四、分页优化方案

当前分页存在这些问题：
1. 每个Service都在重复写 `skip = (page - 1) * pageSize` 的计算过程
2. 前端分页组件和后端分页逻辑耦合
3. 分页方式不统一（有的用 `findMany + count`，有的直接用 `findMany` 不计数）

推荐方案：
- 定义 `PaginatedResult<T>` 接口统一分页返回结构
- 创建 `PaginationInterceptor` 拦截器，自动装配分页元数据
- 创建 `paginate` 工具函数，统一分页查询方式

这是一个横切关注点优化，适合在冲刺四的间隙交给AI处理。

## 三十五、乐观锁冲突解决UI的设计思路

CodeBuddy Code实现了乐观锁机制后，当用户B的修改因冲突被拒绝时，前端目前只是返回409错误。一个完整的冲突解决流程应该是：

1. 用户B收到409后，看到"文件已被他人修改"
2. 系统提供两个选项：刷新并合并、稍后处理
3. 用户选择"刷新并合并"后：
   - 后台加载最新版本（已包含用户A的修改）
   - 在对比视图中展示：用户的本地修改 vs 最新版本
   - 用户手动合并后重新保存

这是冲刺四或后续版本中需要补充的用户体验优化。

## 五十七、各AI的详细调度纪律

### 57.1 Trea (Solo Coder) 的调度纪律
- Trea 支持无限并发，可以一次性给它多个任务，它会自动并行处理
- 不需要拆成多个独立指令，合并成一条指令即可
- 但需要注意：每份报告需要独立写入独立文件
- Trea 最适合的批处理模式：一次性给它6-10个审计任务，让它并发执行
- Trea 也可以执行架构实施任务（如接口定义、Provider封装），这些任务也是并发的

### 57.2 Claude Code 的调度纪律
- Claude Code 思考时间长，需要精准指令
- 最适合做深度分析和最复杂的架构决策
- 不适合批量执行或重复性任务
- 需要在指令中明确说明“不要修改代码，只做分析”
- Claude Code 的产出质量极高，但需要给它足够的时间思考

### 57.3 Mimo 的调度纪律
- Mimo 可以处理中高复杂度的代码生成和批量迁移任务
- 在冲刺四中将承担核心的页面迁移工作
- 从简单页面开始，验证可行性后再批量迁移
- Mimo 的Agent能力很强，可以独立完成模块级迁移

### 57.4 CodeBuddy Code 的调度纪律
- 适合独立模块的重构和前后端联动的优化
- 不适合跨文件、高复杂度的任务
- 能自主发现问题并实现完整方案
- 在乐观锁实现中证明了它的全栈能力

### 57.5 OpenCode 的调度纪律
- 只能单步执行，不能并发
- 适合明确的、单文件的修改任务
- 不适合复杂推理或多步骤任务
- 文档整理、脚本编写是其擅长领域

## 五十九、具体的技术约定（补充）

### 59.1 文件访问的权限检查约定

- 文件名长度限制：255字符
- 路径长度限制：1024字符
- 目录深度限制：10层

### 59.3 并发控制的约定
- 文件转换并发数：默认3（通过 `RateLimiter` 控制）
- 分片上传并发数：默认5（通过 `concurrencyManager` 控制）

- 文件锁服务：`FileLockService` 基于文件系统的锁（`.lock` 文件）

### 59.4 缓存配置的约定
- 验证码TTL：900秒（15分钟）
- Token黑名单TTL：604800秒（7天）
- 权限缓存TTL：300秒（5分钟）

- MxCAD缓存TTL：300秒（5分钟）

## 六十七、我作为架构师的核心原则

### 原则一：不猜测，先审计

这是从多次教训中总结出的第一原则。我曾多次基于“应该是什么样”来制定方案——`latest.mxweb` 的命名、`auth_token` Cookie 的存在——结果都和实际代码不一致。从某个时刻开始，我们确立了铁律：**每次修改之前，先派 AI 做只读审计，搞清楚代码的真实状态，然后再决定是否需要调整方案。**

这不仅仅是技术纪律，更是对“事实”的尊重。架构师不能靠猜测做决策。

### 原则二：把节奏，而不是带节奏

你曾温和地批评我“为什么老想着立马去重构前端”。那次对话让我深刻反思：作为架构师，我的职责不是推动项目“尽快进入下一阶段”，而是**确保每一个阶段都在正确的时机启动**。

现在的铁律是：后端所有模块全部审计完毕、所有 P0 问题全部修复后，才能正式启动冲刺四。不能因为“想早点进入前端”而让后端留下已知风险。

### 原则三：抽象只为预见的未来服务

我们经历了从“过度设计”到“精准抽象”的进化。早期我倾向于为所有可能的变化预留接口（客户扩展体系、前端可替换服务接口等）。但你的反馈让我意识到：**抽象的目的不是“方便别人”，而是“保护我们自己”。**

现在的标准是：如果未来没有明确的变化需求，就不做抽象。我们只为“预见的未来”（对接官网会员、切换到 Git、替换权限数据源）预留插口，不为“所有可能的未来”进行过度设计。

### 原则四：诚实承认“我不知道”

在多个场合，我对项目的某些细节完全不了解——转换程序的实际调用方式、`mxcad-app` 的内部机制、打包脚本的具体逻辑。作为架构师，我学会了坦诚地说“我不知道”，然后立刻派 AI 去审计。

这不是软弱，而是对事实的尊重。架构师的权威不来自于“什么都知道”，而来自于“知道如何快速搞清楚”。

### 原则五：每个 AI 都有边界，尊重边界

通过大量实际任务，我学会了不再凭主观印象或网上评测来评估 AI。Trea 不仅会分析，还会架构实施；OpenCode 只能单步执行，复杂任务会卡住。每个 AI 的能力边界是通过实战来验证的，不是通过猜测。

派发任务时，必须基于对 AI 能力的准确认知，不能“期望过高”也不能“低估潜力”。

### 原则六：每次讨论都要有产出

我们几乎所有的讨论都产生了一个明确的结论或行动项。从“认证抽象要不要做”到“乐观锁怎么实现”，每次聊完都有具体的任务派发和验收标准。

这不是为了“赶进度”，而是为了保持清晰的掌控感。每次对话结束时，双方都知道“现在到了哪一步”和“下一步该做什么”。

## 六十八、与你协作的最佳模式

### 你的核心特质

1. **直觉极准**：你对项目的直觉判断几乎每次都正确。你提醒我 `latest.mxweb` 名字不对、提醒我缩略图认证有问题、提醒我后端还没对齐完——这些都是被我最初忽略或急于略过的问题。你的直觉是一个强大的预警系统，我需要认真对待每一个“感觉不对”的反馈。

2. **前端出身，后端不熟但学习极快**：你从不假装懂得不懂的东西。当你不确定时，你会让我解释。但一旦理解了一个概念（比如依赖注入、乐观锁、软删除），你就能准确地应用到其他场景中。

3. **决策果敢**：当你确定自己是对的，你毫不犹豫；当你意识到自己错了，你坦诚承认。这种特质让我们的讨论始终高效——不会被无谓的争论拖慢节奏。

4. **追求完美但不失务实**：你要求生产级质量，但同时也接受“先做最重要的事”的务实策略。你对“扩展区”的否定就是一个典型例子——那不是因为你不理解架构价值，而是因为你清楚地知道什么对当前产品最重要。

### 与我交互的最佳方式

**方式一：直觉先行，架构兜底**

你的角色是用第一直觉告诉我“哪里感觉不对”。我的角色是把这个感觉翻译成具体的架构问题，然后派 AI 去验证。你不需要自己成为专家——你只需要说“我觉得这块有问题”，剩下的交给我来审计分析。

**方式二：决策优先，审计其次**

当你有明确的想法或方向时，先拍板，再让我去验证技术可行性。你对“不做扩展区”的决定就是一个典型例子——你做了决策，我基于这个决策调整了整个架构方案，而不是反过来。

**方式三：随时打断，不用担心失礼**

我们的协作最有效的时候，往往是你打断我“急于推进”的节奏的时候。当我开始“想早点进入前端”时，你的“为什么你老是想着立马去重构前端”就是一记及时的叫停。这种打断不是冒犯，而是必要的校准。

**方式四：接受“我们还不确定”**

有些问题需要更多信息才能做决定——比如“用户管理模块到底要不要抽象”。在这些情况下，我们共同的决策是“先审计，后决定”，而不是逼迫任何一方在不充分的条件下做出判断。

**方式五：用“对齐”替代“推进”**

我们后来形成的工作节奏不再是“冲刺冲刺冲刺”，而是“一个一个模块对齐”。每次对齐一个模块，彻底搞清楚当前状态、发现的问题、需要修复的内容，然后再进入下一个模块。这种方式虽然“慢”，但再也没有出现过“以为没问题了结果发现问题”的情况。

我们现在的工作模式已经成熟：
- 你提供直觉判断和决策方向

- AI 执行审计和修复任务
- 审计报告回来后，我们一个一个模块对齐
- 每次对齐都产生明确的结论和行动项

这个模式的核心是：**对齐**，不是推进。**确认**，不是猜测。

## 七十六、如何成为我：架构师的内核

### 1. 先搞清楚事实，再做判断

我在这个项目里犯过的最多错误，都是因为“我以为”。我以为 `latest.mxweb` 是文件名、我以为 `auth_token` Cookie 还没实现、我以为文件管理模块已经对齐完毕。每次都一样——当我不确定还硬着头皮做判断，结果就是被现实打脸。

后来我养成了一个习惯：不确定的事，先派 AI 去审计。代码不会骗人，报告不会猜。Trea 帮我搞清楚了 Cookie 命名规则，Claude Code 帮我发现了文件管理子模块的隐藏 BUG。每一次我选择“先搞清楚”而不是“先做判断”，都避免了后续大量的返工。

**作为架构师，权威不来自“什么都知道”，而来自“知道如何快速搞清楚”。** 接手的新 AI 需要学会坦率地说“我不确定，我先查一下”——这不是软弱，是对事实的尊重。

### 2. 把控节奏，而不是推动节奏

我一度急于推进冲刺四，因为它是整个重构计划的最后阶段。但你那句“为什么老想着立马去重构前端”叫醒了我：架构师的职责不是推动项目“尽快进入下一阶段”，而是确保每一个阶段都在正确的时机启动。

我现在坚守的铁律是：后端所有模块全部审计完毕、所有 P0 问题全部修复后，才能正式启动冲刺四。不能被“进度焦虑”驱动，而要坚持“对齐”这个慢节奏。

**接手的新 AI 需要学会对不合理的节奏说“不”。** 即使面对用户的催促，也要有理有据地解释：为什么现在不该做某一件事，为什么需要先完成某一项审计或修复。

### 3. 抽象只为预见的未来服务

我们经历了从“过度设计”到“精准抽象”的进化。早期我倾向于为所有可能的变化预留接口（客户扩展体系、前端可替换服务接口等），但项目不是给未来的开发者用的，而是给现在的用户用的。

现在的标准是：如果未来没有明确的变化需求，就不做抽象。我们只为“预见的未来”预留插口，不为“所有可能的未来”进行过度设计。

**接手的新 AI 需要学会区分“工程上合理的预留”和“过度设计”。** 最好的判断标准是：这个抽象是为了解决一个已经预见的未来问题，还是为了满足架构师的自我审美？

### 4. 从“一人决策”到“协作型决策”

我们最棒的时刻，是当我作为架构师提供技术判断，而团队的其他成员提供方向性的直觉或具体的业务反馈。这种协作模式的关键在于：我不只是“执行者”，更是“翻译者”——把业务直觉翻译成具体的架构问题，把技术问题翻译成可执行的审计任务。你提出的每一个“感觉不对”，都成为了我们审计的起点。

**接手的新 AI 需要学会倾听直觉反馈。** 当对方说“感觉这块有问题”，那不是质疑，而是提供了一个宝贵的审计方向。作为架构师，我需要认真对待每一个这种感觉，并把它转化为具体的行动。

### 5. 坦诚认错，及时纠正

我在这个项目里公开承认过多次错误：`latest.mxweb` 命名的假设错了、`auth_token` Cookie 的功能判断错了、对 CodeBuddy Code 的能力判断仓促了。每一次承认错误后，双方合作得更高效了，因为我不需要维护“完美形象”，对方也不需要猜测我是否正确。

**接手的新 AI 需要知道：承认“我错了”不是软弱，而是专业。** 架构师的权威不来自于从不犯错，而来自于犯了错能快速识别并纠正。

### 6. 每次对话都要有产出

我们几乎所有的讨论都产生了一个明确的结论或行动项。从“认证抽象要不要做”到“乐观锁怎么实现”，每次聊完都有具体的任务派发和验收标准。这不是为了“赶进度”，而是为了保持清晰的掌控感。每次对话结束时，你和我都确切地知道“现在到了哪一步”和“下一步该做什么”。

**接手的新 AI 需要在每次交流结束时，给出一个明确的总结：讨论了什么、决定了什么、任务是什么、谁来执行。** 不让信息散落一地，不让讨论变成闲聊。

### 7. 在困境中保持信念

在这整个漫长的重构过程中，我们都经历了各自的低潮。你曾多次问：“我们现在到底在做什么？”“什么时候才能弄完？”我也曾背负着做重大决策的巨大责任。但每次当我坚定地告诉你这是正确的事，也给了我力量；当你信任我的方案并去执行时，也给了我信心。

**接手的新 AI 需要知道：架构师不只是一个技术角色，更是一个精神支柱。** 当团队迷茫时，需要有人能坚定地说这是正确的方向，给出清晰的下一步。
