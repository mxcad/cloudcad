# 架构约束

> SDD 架构约束文档 — 核心链路的非功能性需求和边界条件
> 关联: [DDD 上下文映射图](../ddd/context-map.md)
> 关联 Feature: [转换图纸](./feature-convert-drawing.feature) | [保存图纸](./feature-save-drawing.feature)

## 性能约束

| 约束 | 阈值 | 说明 |
|------|------|------|
| 保存响应时间 | < 30 秒 | 含 mxweb 写入 + SVN 提交 + bin 生成 |
| 转换响应时间 | < 60 秒 | dwg/dxf → mxweb 格式转换 |
| 大文件支持 | 500MB | 单文件上传上限 |
| 分片上传 | 100 片 | 大文件通过 TUS 协议分片 |
| 并发上传检测 | 同 hash 互斥 | 同一文件 hash 同时只能有一个合并操作 |

## 存储约束

| 约束 | 值 | 说明 |
|------|------|------|
| 核心流通格式 | mxweb | 编辑和存储的唯一内部格式 |
| 支持上传格式 | dwg, dxf, mxweb | 仅这三种 CAD 格式 |
| 非 CAD 文件 | 外部参照专项 | 图片等走外部参照目录，不纳入 FileNode 体系 |
| 存储回滚 | 必做 | SVN 提交失败时，已写入的 mxweb 必须回滚到旧版本 |

## 安全约束

| 约束 | 说明 |
|------|------|
| 保存权限 | 需要 `CAD_SAVE` 项目权限 |
| 上传权限 | 需要 `FILE_CREATE` 项目权限 |
| 输入验证 | 文件格式、大小、项目归属全部校验 |
| 错误信息 | 不泄露内部路径和堆栈信息 |

## 版本约束

| 约束 | 说明 |
|------|------|
| 提交原子性 | mxweb 写入 + SVN 提交 + bin 生成必须是事务性操作 |
| 分片上传 SVN | 暂不提交 SVN，待合并转换成功后统一提交 |
| Library 跳过 SVN | 资源库文件不进行版本管理 |
| 版本号格式 | 由 SVN 自动生成递增版本号 |

## 容错约束

| 约束 | 说明 |
|------|------|
| SVN 提交失败回滚 | 磁盘写入成功后 SVN 失败，必须回滚已写入的文件 |
| 转换引擎故障 | 不创建 FileNode，不写入存储，不提交 SVN |
| 磁盘写满 | 返回 507，回滚已创建的 FileNode |
| 并发保存冲突 | 当前迭代暂不处理，后续通过"拉取最新→合并→重新保存"模式 |

## 接口约束

| 约束 | 说明 |
|------|------|
| API 响应格式 | 统一使用 `ApiResponse<T>` 信封 |
| 错误码语义 | 400(格式)、403(权限)、404(不存在)、422(转换失败)、500(SVN失败)、507(空间不足) |
| 认证 | 所有写操作需要有效 JWT Token |

## 技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| BDD 执行工具 | supertest 直击 NestJS | 零新依赖，团队熟悉，只测 API 层 |
| Gherkin 格式 | `.feature` 文件（直接撰写，不用 Cucumber 解析器） | 作为规格文档，不引入新框架 |
| 领域事件实现 | 先保持直接服务调用 | 降低风险，BDD 验证行为后再重构为事件驱动 |
| 并发保存推迟 | 后续迭代 | 当前代码无此逻辑，先建立基本行为测试 |
