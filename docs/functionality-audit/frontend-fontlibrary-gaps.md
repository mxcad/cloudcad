# FontLibrary 页面行为差异报告

## 对比版本
- **main**：重构前基准（行为真相，代码质量较差但功能完整）
- **refactor/circular-deps**：重构后分支（代码质量改善，功能实现方式改变）

## 总体结论
当前分支功能与 main 分支**基本一致**，核心用户流程（查看、搜索、筛选、排序、上传、删除、批量删除）均已实现。主要差异体现在 API 调用层的封装方式（从自定义 `fontsApi` 迁移到代码生成的 `@/api-sdk` 控制器函数），这是正向重构。但存在一个**功能性 bug**：

### ❌ 缺失/损坏功能

| 功能 | main 状态 | 当前分支状态 | 严重程度 |
|------|-----------|--------------|----------|
| 字体下载 | 可用 | **损坏**（下载请求无法正确转换为 Blob） | 高 |

**问题详情**：  
`handleDownload` 函数中，`fontsControllerDownloadFont` 返回的 `response` 是 `AxiosResponse` 类型，但代码错误地调用了 `response.blob()` 方法（该方法存在于 Fetch API 的 `Response` 对象，不存在于 Axios 响应中）。正确的处理应该是从 `response.data` 中获取 Blob 数据。

**修复方案**：  
将下载逻辑改为与 main 分支一致的方式：直接使用 `response.data` 作为 Blob 数据，或者适当适配 Axios 响应结构。

## 详细对比

### 1. 布局与 UI
- ✅ 无变化：页面头部、统计卡片、标签页切换、筛选栏、视图切换（网格/列表）、批量操作栏、排序选项、底部统计均保持一致。
- ✅ 样式类名和 Tailwind 配置相同，视觉效果一致。

### 2. 字体列表展示
- ✅ 网格视图：卡片布局、图标颜色、文件信息、选择框、悬浮操作按钮均一致。
- ✅ 列表视图：表格列定义、行样式、字体信息（含创建者字段）均一致。
- ✅ 空状态和加载骨架屏展示相同。

### 3. 上传功能
- ✅ 上传模态框的 UI、文件拖拽、类型/大小验证、目标选择（后端/前端/同时）均一致。
- ✅ API 调用从 `fontsApi.uploadFont` 迁移到 `fontsControllerUploadFont`，功能等价。

### 4. 删除功能
- ✅ 单删和批量删除均正常工作，权限检查（`canDeleteFonts`）完整。
- ✅ API 调用迁移到 `fontsControllerDeleteFont`，参数映射正确（`fileName`, `target`）。

### 5. 权限控制
- ✅ 所有权限检查点（读、上传、删除、下载）均使用相同的 `SystemPermission` 常量，逻辑一致。

### 6. 搜索与筛选
- ✅ 按名称搜索、按扩展名筛选、按日期范围筛选均正确实现，前端筛选逻辑与 main 相同。
- ✅ 筛选条件重置、展开/收起筛选面板行为一致。

### 7. 排序
- ✅ 按名称、大小、修改时间排序，升序/降序切换与 main 一致。

### 8. 错误处理
- ✅ 所有 API 调用均有 try-catch，失败时通过 `showToast` 提示用户，与 main 行为相同。

### 9. 其他注意事项
- 分页功能在 main 和当前分支均**缺失**（一次性加载所有字体），这不是本次重构引入的差异，是否增加分页需产品/技术决策。
- 当前分支在 API 响应解析上更严谨（处理了 `{ data: { data: [...] } }` 嵌套结构），比 main 分支的简单解析更健壮。

## 待决策项
- **分页支持**：字体库可能包含大量字体（尤其 SHX 字体），当前无分页可能导致性能问题和糟糕的加载体验。建议后端提供分页参数，前端增加分页组件。
- **上传目标“同时上传”**：该选项在 main 分支也存在，但实际行为需要后端支持将文件同时写入两个存储位置。若后端尚未实现，该选项可能无效，需确认。

## 修复计划
1. **立即修复下载功能**（已在本轮对话中完成）。
2. 其他差异无需改动，重构后的代码质量优于 main 分支。
