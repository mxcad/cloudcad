# 功能对齐 — 核心方案

> 版本: 2.0
> 日期: 2026-05-08

---

## 一、核心理念

### 1.1 什么是"对齐"

main 分支的代码实现很烂，但它的**用户可见行为**是完整且正确的。

当前分支的实现更优，但缺失了很多功能、交互、样式。

**对齐 = 让当前分支的用户行为与 main 完全一致，同时保留更好的实现方式。**

### 1.2 AI 的自由度

AI 需要：

1. **理解意图**：main 上这个功能的目的是什么？用户期望看到什么？
2. **判断现状**：当前是"完全缺失"、"部分实现"、"实现方式不同但结果一致"、还是"引入了 bug"？
3. **选择策略**：
   - 当前实现更好且行为正确 → **保持现状**
   - 当前实现更好但行为有偏差 → **在当前实现基础上修正**
   - 当前实现缺失了功能 → **用当前架构重新实现**
   - 当前实现引入了 bug → **修复 bug，保留新架构**
   - main 的某个行为是过时的/hack → **标记"不建议恢复"，等用户决策**

### 1.3 绝对不能做的事

- 把代码改回 main 的实现方式（`as any`、硬编码等）
- 为了"一致性"而降低代码质量
- 在不确定的情况下删除当前分支的功能
- 跨模块修改文件（发现跨模块影响时停下来写文档）

## 三、任务分类

| 类型 | 处理方式 | 判断标准 |
|------|----------|----------|
| **简单修复** | agent 直接修改 + commit | 单模块、不跨域、改动明确 |
| **复杂修复** | 写入文档 + 等用户决策 | 跨模块、需业务判断、不确定 |
| **深度文档** | 持续生成 | 架构图、流程图、决策树 |

## 四、输出格式

### 差异报告（写入 docs/functionality-audit/[模块]-gaps.md）

```markdown
# [模块名] — 功能差异报告

## 差异 1: [标题]

### main 行为
[用户看到什么、做到什么、结果是什么]

### 重构后行为
[现在是什么]

### 差异类型
- 功能缺失 / 行为偏差 / 样式偏差 / 交互缺失 / 边界缺失 / main 过时

### 影响范围
- 涉及文件: [列出]
- 影响模块: [列出]

### 修复策略
- 简单修复 → agent 直接 commit
- 复杂修复 → 需要用户决策
- 当前实现更优 → 保持现状
- main 行为过时 → 标记不建议恢复

### 修复建议
[用当前架构怎么实现]

### 需要决策
[如果需要用户判断，说明具体问题]
```

## 五、调度架构

### 永不停歇机制

```
1. 调度者维护任务池
2. agent 完成 → 立即从池中取下一个任务 → 启动新 agent
3. 任务来源：未审计区域 → 新发现问题 → 用户决策后 → 深度文档
4. 任务池不足时 → 启动深度文档 agent（永远做不完）
5. 除非：查无可查、改无可改、写无可写
```

### 冲突避免

```
1. agent 开始前登记要改的文件
2. 调度者检查无冲突后批准
3. 跨模块 → 停止修改，写入文档
```

### 断点续接

**状态持久化文件：** `docs/functionality-audit/DISPATCHER-STATE.md`

调度者每次调度循环结束后，必须更新此文件。内容包括：
- 已完成的任务列表
- 进行中的任务状态
- 待分配任务池
- 待用户决策列表
- 已审计区域进度

**重新启动流程：**

```
1. 读取 DISPATCHER-STATE.md
2. 如果有"进行中的任务" → 用 git log 验证是否已提交
3. 将未完成的任务重新分配
4. 启动 36 个 agent 继续
5. 不要从头开始，不要重复已完成的工作
```

**关键：** 状态文件是防止中断丢失的唯一手段。每完成一个 agent 的调度循环就更新一次。
