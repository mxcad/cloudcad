# 功能对齐 — 核心方案

> 版本: 2.0
> 日期: 2026-05-08

---

## 一、核心理念

### 1.1 什么是"对齐"

对其的是目标。 目标是对齐
main 分支的代码实现很烂，但它的**用户可见行为**是完整且正确的。

当前分支的实现更优，但缺失了很多功能、交互、样式。

**对齐 = 让当前分支的用户行为与 main 完全一致，同时保留更好的实现方式。**

### 1.2 AI 的自由度

AI 需要：

1. **理解意图**：main 上这个功能的目的是什么？用户期望看到什么？
2. **判断现状**：当前是"完全缺失"、"部分实现"、"实现方式不同但结果一致"、还是"引入了 bug"？
3. **选择策略**：
   - 当前实现更好且行为正确 → **保持现状**
   - 当前实现更好但行为有偏差 → **在当前实现基础上修正**
   - 当前实现缺失了功能 → **用当前架构重新实现**
   - 当前实现引入了 bug → **修复 bug，保留新架构**
   - 剩余情况请自行判断决定
### 1.3 绝对不能做的事

- 把代码改回 main 的实现方式（`as any`、硬编码等）
- 为了"一致性"而降低代码质量
- 在不确定的情况下删除当前分支的功能

## 二、对比维度清单

### 2.1 API 层
- 路径、HTTP 方法、请求参数、响应结构、状态码、错误格式、权限要求

### 2.2 前端行为
- 路由、布局、表单字段、校验规则、按钮行为、加载/错误/空状态、权限处理

### 2.3 样式
- 布局结构、间距、颜色、字体、响应式断点、动画、暗色模式

### 2.4 业务流程
- 完整操作链路、分支条件、并发/竞态、离线/网络异常、多标签页同步

### 2.5 其他
- 只要是和我们最终目标一致的其他清单
- 比如没有其他任务了，可以安排真正有价值的测试覆盖等等。

## 三、任务重量控制

**核心原则：每个 agent 的任务必须能在合理时间内完成。**

确保36个子代理的快速的调度，并行运行，始终保持36个子代理。所以你需要不断运行，不断用一部分子代理去找任务/拆分任务/审查任务/修改任务等等，你去调度任务。

---

## 四、任务分类

与最终目标一致的所有任务

---

## 四、输出格式

需要梳理出每个功能的流程图/架构图/逻辑图/用户交互操作/权限等等值得持久化保存的文档。

所以36个子代理始终并行运行，你可以利用无任务的情况下派发大量的子代理补充。当然先以目标任务为主。
---

## 五、调度架构

### 永不停歇机制

```
1. 调度者维护任务池
2. agent 完成 → 立即 启动新 agent 分配任务
3. 任务来源：子代理根据目标对齐，找的任务
4. 任务池不足时 → 启动文档输出 agent（永远做不完）
5. 除非：查无可查、改无可改、写无可写
```

## 六、质量保证

- agent 修复后验证 TypeScript 编译
- API 变更时同步更新前端 SDK 类型
- 最终验收：`pnpm type-check` + `pnpm build`
