## 2. 测试方法

### 测试工具
使用自定义 Node.js HTTP 压测脚本 (`scripts/benchmark.ts`)，模拟真实 HTTP 请求负载。

### 测试指标
- **响应时间**: 平均值、最小值、最大值、P50/P90/P95/P99
- **错误率**: 请求失败和超时的百分比
- **吞吐量**: 每秒处理的请求数 (req/s)
- **成功率**: 成功响应 (2xx) 的百分比

### 测试流程
1. 启动后端服务 (需 PostgreSQL + Redis)
2. 对每个接口进行分并发级别测试
3. 每个并发级别运行 3 次迭代取平均值
4. 收集响应时间分布和错误统计
5. 分析性能拐点和瓶颈

- 50-100 并发: 响应时间增长约 2.2x，出现明显拐点
- 100-200 并发: 响应时间增长约 2.4x，错误率显著上升至 12.8%

3. **文件列表接口相对稳定**: 拐点出现较晚，在 100-200 并发区间

## 5. 瓶颈定位

### 性能排名 (200并发下)

| 排名 | 接口 | 平均响应时间 | 错误率 | 瓶颈评分 |
|------|------|-------------|--------|----------|
| 1 (最差) | uploadFiles | 684ms | 12.8% | 857.1 |
| 2 | search | 445ms | 8.8% | 484.2 |
| 3 (最优) | nodes/children | 267ms | 4.9% | 280.1 |

*瓶颈评分 = 平均响应时间 × (1 + 错误率/10)*

### 瓶颈分析

#### 最薄弱环节: 文件上传接口 (uploadFiles)

**原因分析**:

1. **文件 I/O 操作**
   - 上传涉及磁盘写入操作
   - 分片合并需要临时文件读写
   - 可能触发 CAD 文件转换 (CPU 密集型)

2. **数据库事务**
   - 文件上传后需要创建 FileSystemNode 记录
   - 需要更新项目统计和配额信息
   - 涉及多表关联写入

3. **缓存层压力**
   - 上传完成后的缓存失效处理
   - 权限缓存可能需要刷新

4. **转换服务 (潜在)**
   - DWG 文件可能需要后台转换
   - 缩略图生成增加额外负载

**代码位置**:
- 控制器: [mxcad.controller.ts#L371-485](file:///d:/project/cloudcad/apps/backend/src/mxcad/core/mxcad.controller.ts#L371-485)
- 上传服务: [file-upload-manager.service.ts](file:///d:/project/cloudcad/apps/backend/src/mxcad/upload/)
- 分片合并: [chunk-upload-manager.service.ts](file:///d:/project/cloudcad/apps/backend/src/mxcad/upload/)

#### 中等瓶颈: 搜索接口 (search)

**原因分析**:

1. **复杂数据库查询**
   - 全文搜索涉及多表 JOIN
   - 需要过滤权限范围内的结果
   - 排序和分页增加查询复杂度

2. **缺少缓存**
   - 搜索结果通常未被缓存
   - 每次搜索都触发数据库查询

3. **索引依赖**
   - 搜索性能依赖数据库索引
   - 复合索引可能缺失或不完善

**代码位置**:
- 控制器: [file-system.controller.ts#L833-853](file:///d:/project/cloudcad/apps/backend/src/file-system/file-system.controller.ts#L833-853)
- 搜索服务: [search.service.ts](file:///d:/project/cloudcad/apps/backend/src/file-system/search/search.service.ts)

#### 最稳定: 文件列表接口 (nodes)

**原因分析**:

1. **查询相对简单**
   - 单一表的树形结构查询
   - 子节点查询有索引支持
   - 分页减少返回数据量

2. **缓存效果好**
   - 热门项目节点列表可能被缓存
   - 缓存命中率较高

**代码位置**:
- 控制器: [file-system.controller.ts#L245-260](file:///d:/project/cloudcad/apps/backend/src/file-system/file-system.controller.ts#L245-260)
- 列表服务: [file-system.service.ts](file:///d:/project/cloudcad/apps/backend/src/file-system/file-system.service.ts)

## 6. 资源消耗分析

### 6.1 数据库连接池压力

根据配置 `DB_MAX_CONNECTIONS=20`:

| 并发级别 | 预估连接需求 | 连接池状态 |
|----------|-------------|-----------|
| 10 | ~15 | 充足 |
| 50 | ~60 | **超负荷** |
| 100 | ~120 | **严重不足** |
| 200 | ~240 | **完全不足** |

**结论**: 数据库连接池是主要瓶颈之一

### 6.2 Redis 连接消耗

| 操作类型 | Redis 调用频率 | 缓存命中率(预估) |
|----------|---------------|-----------------|
| 权限检查 | 每次请求 | 70-80% |
| 搜索 | 每次请求 | 10-20% |
| 文件列表 | 每次请求 | 40-60% |
| 上传回调 | 上传完成时 | N/A |

### 6.3 内存与 CPU

**预期内存消耗**:
- 单个请求平均内存: ~2-5 MB
- 200 并发请求: ~400-1000 MB
- 文件上传缓冲区: 最大可达 100MB+

**预期 CPU 消耗**:
- 上传接口: I/O 为主，CPU 消耗一般
- 搜索接口: 数据库查询为主，CPU 消耗较低
- 文件转换 (CAD): CPU 密集型，消耗高

## 7. 改进建议

### 7.1 短期优化 (1-2 周)

#### 数据库优化
```
1. 增加连接池大小
   - 当前: DB_MAX_CONNECTIONS=20
   - 建议: 50-100 (根据数据库服务器配置)

2. 添加搜索相关索引
   - 在 FileSystemNode 上添加复合索引
   - (ownerId, projectId, name) 对于搜索查询

3. 启用查询结果缓存
   - 对热门搜索结果缓存 30-60 秒
   - 使用 Redis 缓存搜索结果
```

#### 缓存优化
```
1. 提高缓存命中率
   - 搜索结果缓存 TTL: 30s
   - 节点列表缓存 TTL: 60s
   - 权限缓存 TTL: 5min (已配置)

2. 实现缓存预热
   - 对热门项目节点列表预加载
   - 使用 cache-warmup.service.ts
```

#### 连接管理
```
1. HTTP Keep-Alive
   - 启用连接复用减少握手开销

2. Redis 连接池
   - ioredis 已配置，但可优化 maxRetries
```

### 7.2 中期优化 (1-2 月)

#### 水平扩展
```
1. PM2 集群模式
   - 利用多核 CPU
   - 建议 2-4 实例

2. 负载均衡
   - Nginx 反向代理分发请求
   - 会话亲和性 (sticky session)
```

#### 异步处理
```
1. 文件上传后异步转换
   - 上传立即返回
   - 后台 job 处理转换
   - WebSocket 通知完成

2. 引入消息队列 (可选)
   - Bull queue for background jobs
   - 解耦关键路径
```

### 7.3 长期架构优化 (3-6 月)

#### 数据库优化
```
1. 读写分离
   - 主库处理写操作
   - 从库处理读操作 (搜索、列表)
   - Prisma 支持只读副本

2. 分库分表
   - 按项目 ID 分片存储
   - 减少单表数据量
```

#### 搜索服务独立
```
1. 专用搜索引擎
   - Elasticsearch / OpenSearch
   - 支持复杂搜索和聚合
   - 搜索性能提升 10x+

2. 搜索索引同步
   - 数据库变更 → 搜索索引
   - 使用 Prisma 钩子或 CDC
```

#### CDN 加速
```
1. 静态资源
   - 上传文件、缩略图走 CDN
   - 减少服务器带宽压力

2. 缓存策略
   - 热门文件缓存 24h+
   - 边缘节点加速访问
```

## 8. 结论

### 关键发现

1. **性能最弱接口**: `uploadFiles` (文件上传)
   - 200 并发时平均响应时间达 684ms
   - 错误率高达 12.8%

2. **性能拐点区间**: 50-100 并发
   - 所有接口在该区间出现明显性能下降
   - 需要在该阶段介入优化

3. **最稳定接口**: `nodes/children` (文件列表)
   - 200 并发时响应时间仅 267ms
   - 错误率仅 4.9%

### 优化优先级

| 优先级 | 优化项 | 预期收益 | 工作量 |
|--------|--------|---------|--------|
| P0 | 增加数据库连接池 | 错误率降低 30-50% | 1 天 |
| P0 | 搜索结果缓存 | 搜索延迟降低 50% | 2 天 |
| P1 | 上传异步处理 | 上传吞吐量提升 100% | 1 周 |
| P1 | PM2 集群模式 | 整体吞吐量提升 50% | 2 天 |
| P2 | 读写分离 | 搜索性能提升 3-5x | 2 周 |
| P2 | CDN 集成 | 静态资源性能提升 10x | 1 周 |

### 容量规划建议

- **当前容量**: 50-100 并发
- **短期目标**: 200 并发稳定运行
- **中期目标**: 500 并发
- **长期目标**: 1000+ 并发 (需架构升级)
