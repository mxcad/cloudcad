**日期**: 2026-05-02

**基于**: `docs/sprint3-test-quality-audit.md`（Trea 质量审计）

## 一、风险分级框架

优先级判定基于四个维度加权：

| 维度 | 权重 | 说明 |
|------|------|------|
| **数据安全风险** | 40% | 逻辑缺陷是否会导致数据丢失/泄漏/不可恢复 |
| **业务连续性风险** | 30% | 缺陷是否会导致核心功能不可用 |
| **修复成本递增** | 20% | 现在不改，后续重构时改动成本是否陡增 |
| **测试基础价值** | 10% | 该测试能否为后续测试提供可复用的 mock/工具 |

## 二、三层优先级路线图

### 第一层：数据安全防线（必须立即补）

> 如果只做一件事，就做这个。这些缺陷直接威胁用户数据的完整性和可恢复性。

#### T1 — 引用计数逻辑测试

| 属性 | 内容 |
|------|------|
| **目标服务** | FileOperationsService, FileTreeService, MxCadController |
| **风险** | 两个用户上传了同一个文件（相同哈希），当其中一人删除时，物理文件也被删除，导致另一人的文件变成死链。或者反过来——永远不删，存储泄漏 |
| **现状** | 零覆盖。所有删除/上传测试只用了唯一的哈希值 |
| **为什么排第一** | 这是唯一一个**无论测试怎么跑都发现不了**的缺陷类型——它只在生产环境的特定数据模式下触发。而且后果是**不可逆的数据丢失** |
| **核心场景**（3-5 个） | ① 两个节点引用同一哈希 → 删一个节点 → 验证物理文件仍在；② 最后一个引用也删除 → 验证物理文件被清理；③ 上传重复文件 → 验证不创建重复物理存储、引用计数递增 |
| **预估工作量** | 3-4 小时 |

#### T2 — SVN 提交失败回滚测试

| 属性 | 内容 |
|------|------|
| **目标服务** | VersionControlService |
| **风险** | 提交过程中 SVN 服务器异常（断连、超时、磁盘满），已写入的数据库记录已提交但 SVN 没有对应的版本快照，导致数据库与文件系统状态不一致 |
| **现状** | VersionControlService mock 全部返回 `isReady=false`，即所有 SVN 操作路径从未被触发 |
| **为什么排第二** | 文件操作服务（FileOperationsService）的所有删除/移动逻辑在内部调用 SVN 提交。如果 SVN 回滚逻辑有问题，**整个文件系统的版本历史会损坏**，而这是用户恢复误删文件的最后手段 |
| **核心场景**（3-4 个） | ① SVN commit 中途失败 → 验证数据库事务回滚；② 部分文件提交成功部分失败 → 验证整体回滚或标记不一致；③ 并发提交冲突 → 验证锁重试和最终一致性 |
| **预估工作量** | 2-3 小时 |

### 第二层：核心业务逻辑补全（本周完成）

> 这些测试覆盖现有 270 个用例中"看起来有但实际欠缺"的业务规则验证。

#### T3 — 并发冲突基础测试

| 属性 | 内容 |
|------|------|
| **目标服务** | FileOperationsService, FileTreeService, MxCadController（各 2-3 个场景） |
| **风险** | 用户 A 和用户 B 同时移动/删除同一个节点，或同时上传同一个文件，在没有正确锁机制的情况下会导致数据库不一致或文件系统孤儿节点 |
| **现状** | 270 个测试用例全部是单线程顺序执行 |
| **为什么排第三** | 并发缺陷是典型的"测试环境从不出现、生产环境偶发"类型。冲刺二刚刚完成了上帝模块拆分，服务间调用链变长，**并发窗口被放大** |
| **核心场景**（6-9 个） | ① 同时删除同一节点 → 验证一人成功一人收到冲突错误；② 同时移动同一节点到不同目标 → 验证最终状态一致；③ 同时上传相同哈希文件 → 验证不创建重复节点 |
| **预估工作量** | 3-4 小时 |

#### T4 — 完成已有 spec 骨架的 TODO 实现

| 属性 | 内容 |
|------|------|
| **目标服务** | ProjectCrudService（spec 存在，TODO）、FileConversionService（spec 存在，TODO） |
| **风险** | 冲刺二已将这两个服务拆分为独立模块，但没有有效测试保护，后续重构容易引入回归 |
| **为什么排第四** | 这两个服务的测试骨架已经搭建好了（mock 配置、describe 块就绪），只是断言是空的。**填充比从零创建快得多**——性价比最高 |

| **预估工作量** | 3-4 小时 |

### 第三层：P0 核心 Service 测试基础（本周到下周）

> 从零开始为三个完全没有测试的大 Service 建立测试防线。

#### T5 — UsersService 单元测试

| 属性 | 内容 |
|------|------|
| **目标服务** | `users/users.service.ts`（1141 行） |
| **为什么是三个 P0 中最优先的** | UsersService 是**所有其他服务的上游依赖**——AuthFacadeService、InitializationService、ProjectMemberService 都调用它。UsersService 的 `create()` 方法在一个事务中同时创建用户和私人空间，逻辑最复杂、影响面最广 |
| **核心场景** | ① 创建用户（含私人空间创建的事务性）；② 邮箱/手机号唯一性冲突；③ 软删除→冷静期→物理删除的完整生命周期；④ 账户注销的多验证方式（密码/短信/邮箱/微信）；⑤ 密码修改（有密码用户 vs 无密码用户的差异逻辑） |
| **预估工作量** | 3-4 小时 |

#### T6 — AuthFacadeService 单元测试

| 属性 | 内容 |
|------|------|
| **目标服务** | `auth/auth-facade.service.ts`（1066 行） |
| **为什么比 MxCadService 优先** | 认证是系统的入口。如果认证逻辑有缺陷，要么安全漏洞（越权登录），要么功能不可用（所有用户被锁在外面）。而且 AuthFacadeService 的 Facade 模式意味着它聚合了 6+ 个子服务的调用——是一个**集成风险集中点** |
| **核心场景** | ① 登录（邮箱/用户名/手机号三种方式 + 密码验证）；② 注册（邮箱验证开关、重复检测）；③ 微信登录/绑定/解绑流程；④ Token 刷新和黑名单；⑤ 注册开关的运行时配置控制 |
| **预估工作量** | 3-4 小时 |

#### T7 — MxCadService 单元测试

| 属性 | 内容 |
|------|------|
| **目标服务** | `mxcad/core/mxcad.service.ts`（890 行） |
| **为什么排最后** | MxCadService 的复杂度主要在于外部依赖（文件系统、格式转换命令行工具、MxWeb 程序），这些天然难以单元测试。其核心逻辑实际上是委托给子服务（FileUploadManagerFacade、FileSystemNodeService、FileConversionService）——**子服务已有测试覆盖时，MxCadService 的风险相对可控** |
| **核心场景** | ① 分片上传流程（checkChunkExist → uploadChunk → mergeUploadedFile）；② 预加载数据获取；③ 缩略图请求处理；④ 外部参照统计查询；⑤ 文件转换参数传递 |
| **预估工作量** | 3-4 小时 |

## 四、为什么这个顺序

### 4.1 引用计数为什么排第一

引用计数的缺陷有一个特殊属性：**它不是"会不会出问题"，而是"什么时候出问题"**。

- 单元测试跑 100 次可能全绿——因为 mock 数据每次都是用唯一哈希
- 集成测试也发现不了——除非特意构造两条记录共享同一个哈希
- 到了生产环境，用户上传了同一个 .dwg 文件到两个不同的项目，然后删掉其中一个……另一个项目的文件就变成了无法下载的死链

这类缺陷属于**静默数据损坏**——没有报错、没有异常日志、系统看起来一切正常，但用户的数据已经丢了。在软件缺陷分类中，这是严重度最高的一类。

### 4.2 SVN 回滚为什么紧接其后

冲刺二中 FileOperationsService 被拆分为独立模块后，所有文件删除/移动操作都会调用 VersionControlService 来维护 SVN 版本历史。当前的测试全部 mock 了 `isReady=false`，这意味着：

- SVN 集成路径的代码行覆盖率为 **0%**
- SVN commit 调用的 `$transaction` 事务边界从未被验证
- SVN 失败时的数据库回滚逻辑从未被执行过

生产环境中 SVN 服务不是 100% 可用的——网络抖动、磁盘空间不足、权限变更都可能导致提交失败。如果不验证回滚逻辑，一次 SVN 故障就会让文件系统元数据和实际存储产生永久性偏移。

### 4.3 并发测试为什么在 P0 Service 之前

这不是因为并发比 P0 Service 的测试更重要，而是因为**冲刺二刚刚完成上帝模块拆分**。拆分前，所有逻辑在一个巨大的 Service 文件中，NestJS 的单例模式天然提供了一定程度的串行化。拆分后，调用链变成 A 模块 → B 模块 → C 模块，每个模块内部有独立的缓存和状态管理，**并发窗口从毫秒级扩大到秒级**。

这个风险是冲刺二**直接引入的新风险**，应当在新架构稳定之前就验证。如果等到所有 P0 Service 测试写完再回来测并发，可能已经过了几周，架构细节已经被遗忘。

### 4.4 三个 P0 Service 之间的内部排序

- **UsersService 第一**：它是其他服务的数据源。所有涉及用户的测试（认证、权限、成员管理）都间接依赖 UsersService 的行为。如果 UsersService 的测试基础没打好，其他测试的 mock 可能就是错的。
- **AuthFacadeService 第二**：它是系统的安全边界。认证逻辑的错误要么导致功能障碍（所有人登不进去），要么导致安全漏洞（不该登录的人登进去了）。
- **MxCadService 第三**：它的核心逻辑是委托和编排。其子服务（FileConversionService、FileSystemNodeService、ChunkUploadService）要么已有测试，要么在 T4 中补充。当子服务被充分测试后，MxCadService 的残留风险是一层薄薄的编排逻辑——`checkChunkExist` 调了哪个子服务、参数有没有传对。这类逻辑的缺陷通常表现为功能不可用（而非数据损坏），严重度较低。

## 五、不做什么（以及为什么）

| 被推迟的项 | 原因 |
|-----------|------|
| FileMergeService 单元测试（834 行） | 核心逻辑是调用外部命令行工具合并分片文件，难以单元测试；更适合集成测试 |
| FileSystemNodeService 单元测试（768 行） | 内部大量并发控制代码（`creatingNodes` Map），需配合 T3 并发测试一起设计 |
| ProjectMemberService 单元测试（648 行） | 依赖 FilePermissionService + AuditLogService，mock 链长；在 P0 Service 有基础后再做 |
| E2E 关键链路测试（Playwright） | 需要完整测试环境（数据库、Redis、文件存储），基础设施投入大，应在单元测试防线建立后再投入 |
| 循环引用检测测试 | 当前树结构的约束（`parentId`）在数据库层面已有外键约束，循环引用在应用层发生概率低 |
