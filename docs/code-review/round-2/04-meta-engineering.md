## 一、09-typescript-safety.md 元审查

### 1.1 后端 tsconfig 严重偏离 strict 模式

- **原始报告**：§1.2，严重程度 🔴 严重

- **判断理由**：报告正确地识别了后端关闭了 `strictNullChecks`、`noImplicitAny`、`noImplicitReturns`、`strictPropertyInitialization` 等多个关键 strict 选项。报告中指出 "Biome 配置中 `noExplicitAny` 和 `noUnusedVariables` 规则也是禁用的，这意味着实际上没有任何工具在强制执行后端类型安全" 这一结论正确。修复建议"逐步开启 strictNullChecks" 合理，但报告正确承认该修复需要大量改动，没有建议一刀切强制开启。

### 1.2 前端 tsconfig 配置

- **判断理由**：报告仅陈述了前端配置正确继承根 strict 选项的事实，属正面发现。无需修改。

### 1.3 后端生产代码 `Promise<any>` 返回类型

- **原始报告**：§2.2.1，多处 🔴/🟠
- **分类**：**A — 有效且需要修复**
- **判断理由**：`validateUser()` 返回 `Promise<any>` 确实丢失了 Prisma 生成的强类型信息，`IPublicLibraryProvider` 接口全返回 `Promise<any>` 会使接口契约形同虚设。修复建议合理——需要具体类型或至少 `Promise<unknown>`。但需注意：
  - `redis-cache.service.ts` 中的 `get<any>(key)` 是合理的泛型用法，缓存层无法预知值类型，不应标记为问题。此处审查**略微过度**，更合理的分类应为 B，因泛型参数确实无法预先约束。但整体方向正确。

### 1.4 类型声明为 `any` 的变量

- **原始报告**：§2.2.2，🟠 高
- **分类**：**A — 有效且需要修复**
- **判断理由**：`where: any = {...}` 确实应使用 `Prisma.FileSystemNodeWhereInput` 等 Prisma 生成的类型。`storageInfo: any = null` 应定义具体接口。修复建议正确。

### 1.5 `as any` 在生产代码中

- **原始报告**：§2.2.3，🟠/🟡
- **分类**：拆分评估：
  - `user: null as any`（local-auth.provider.ts）：**A** — 确实存在类型安全问题
  - `config as never`（policy-engine）：**B — 无需修复** — 这在策略工厂模式中是常见的类型断言用法，用于下转型到具体策略类型，修复建议中"改用泛型约束"的收益与复杂度不匹配
  - `ensure-libraries.ts:75`：**B — 无需修复** — 报告已标记为 🟢 低，种子脚本中使用 `as any` 可接受

### 1.6 前端 `as unknown as` 模式

- **原始报告**：§2.3.1，🔴/🟠/🟡
- **分类**：**A — 有效且需要修复**

### 1.7 测试代码中的 `as any`

### 1.8 `@ts-ignore` / `@ts-expect-error` 审查

### 1.9 非空断言（`!`）审查

- **原始报告**：§4.1-4.2

  - §4.1.1 配置读取：**A — 有效且需要修复** — 配置缺失时 `!` 断言会导致不清晰的 `TypeError` 崩溃，应替换为带明确错误信息的守卫

### 1.10 泛型使用 & `as never`
