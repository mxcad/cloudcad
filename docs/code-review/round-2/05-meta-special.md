## 二、14-svn-version-control.md 问题逐一评估

### 问题 1.1 — 命令注入风险（字符串拼接构建 shell 命令）

- **分类**：**A（有效）**
- **理由**：代码验证确认。11 个文件使用 `executeCommand()`（底层 `child_process.exec`，通过 shell 执行），仅 `svncommit.js` 和 `svncat.js` 使用 `executeSpawn`/`executeExecFile`（非 shell 参数数组）。`repoUrl`、`targetPath`、`username` 等参数直接字符串拼接，存在真实的命令注入面。
- **攻击面评估**：需确认这些参数的来源。若来自数据库存储的配置（非用户直接输入），风险降级但问题仍存在。建议至少对所有参数加双引号包裹，优先改用 spawn/execFile。

### 问题 1.2 — 密码明文通过命令行传递

- **分类**：**A（有效）**
- **理由**：真实的安全问题。`--password` 明文出现在进程参数中（`ps aux` 可见），`svncheckout.js:17-18` 等多处均存在。即使在 executeSpawn 中也无法避免参数可见问题。修复建议（使用 `--password-from-stdin` 或环境变量）切实可行。

### 问题 1.3 — 缺少子进程超时处理

- **分类**：**A（有效）**
- **理由**：代码验证确认。`svn-executor.js` 中三个执行函数均未设置超时。SVN 操作（checkout、commit、update）可能因网络超时等原因无限挂起。`executeCommand` 底层 `exec` 原生支持 `options.timeout`，修复成本低。

### 问题 1.4 — 临时文件命名碰撞和清理不完整

- **分类**：**B（无需修复）**
- **理由**：使用 `Date.now() + Math.random()` 碰撞概率极低。SIGKILL 场景下任何临时文件机制都无法保证清理（进程被强制终止，无法执行清理代码），这不是可修复的代码缺陷。使用 `fs.mkdtempSync` 会好一些但改善有限。`finally` 模式重构（svncommit.js 中 unlinkSync 重复）是代码风格改进，不影响功能。

### 问题 1.5 — svnpropset 简单属性值未做 shell 转义

- **分类**：**A（有效）**
- **理由**：真实的安全问题。第 32 行仅对双引号做转义，未处理反引号、`$()`、`$var` 等 shell 特殊字符。由于使用 `executeCommand()`（shell 模式），攻击向量成立。修复建议（改用 executeSpawn/executeExecFile 或 stdin）正确。

### 问题 1.6 — svncat Buffer 转换冗余

- **分类**：**B（无需修复）**
- **理由**：审查描述本身承认 `encoding: 'buffer'` 通过 `Object.assign` 可以工作。`Buffer.from(buffer)` 创建副本是冗余但无功能影响，纯代码整洁性问题。建议合并但不值得独立追踪。

### 问题 2.1 — Service 与 Provider 大量代码重复

- **分类**：**D（需用户确认）**
- **理由**：审查描述准确（两个文件确实重复约 800 行），但需确认设计意图。`SvnVersionControlProvider` 实现了 `IVersionControl` 接口（策略模式），可能是为未来支持其他 VCS（如 Git）预留。`VersionControlService` 作为 Facade 层可能有不同的职责边界。不可轻易判断此架构为"错误"，需与团队确认架构演进方向。

### 问题 2.2 — SVN 与数据库操作缺乏事务性保证

- **分类**：**A（有效）**
- **理由**：与报告 16-问题6 发现同一问题。代码验证确认：`permanentlyDeleteNode`（L1405-1474）中数据库事务先于文件/SVN 操作，且 SVN 失败仅 catch 日志不抛出。这是真实的分布式事务问题，修复建议（补偿事务、先 SVN 后数据库）合理。
- **跨报告重叠**：与 16-问题6 重合，但 14-2.2 更侧重于删除流程而非提交流程。

## 四、16-async-concurrency.md 问题逐一评估

### 问题 1 — 文件名唯一性 TOCTOU

- **分类**：**A（有效）**
- **理由**：真实的竞态条件。`checkNameUniqueness` 和 `createFileNode` 之间无事务保护。但需注意：代码中 `createFileNode` 内部有重名处理（生成 `file (1).dwg`），这意味着并发冲突不会导致错误，只会产生带后缀的文件名。严重程度应从"严重"调整为"中"——实际影响是非预期的自动重命名而非数据丢失。
- **跨报告重叠**：与 14-7.1（删除顺序）相关但不重复——属不同操作路径。

### 问题 2 — ConcurrencyManager 非分布式锁

- **分类**：**A（有效）**
- **理由**：代码验证确认。`ConcurrencyManager` 使用内存 `Map`，注释称"提供分布式锁机制"但实际不是。多实例部署时无法跨进程互斥。修复建议（重命名并更新文档）是最小方案，Redis 锁升级是架构改进。

### 问题 3 — storeRefreshToken 非原子操作

- **分类**：**A（有效）**
- **理由**：`deleteMany` + `create` 两步之间崩溃会导致所有设备 refresh token 丢失。`$transaction` 包裹是最直接的修复。这是真实的可用性问题。

### 问题 4 — clearTrash 传入非事务客户端

- **分类**：**A（有效）**
- **理由**：代码验证确认。`clearTrash` 方法 L1631 将 `this.prisma`（非事务客户端）传递给 `deleteFileIfNotReferenced`，而该方法接受 `Prisma.TransactionClient` 类型。虽然 TypeScript 配置 `strictNullChecks: false` 且 `noImplicitAny: false`（允许此调用编译通过），但破坏了接口契约。`performPermanentCleanup` 也同样传入 `this.prisma`。

### 问题 5 — 批量操作无事务保护

- **分类**：**D（需用户确认）**
- **理由**：审查准确描述 `for...of` 串行执行 + 独立事务的原子性缺失。但 `Promise.allSettled` 修复建议改变了 API 语义（从串行变并行），且 Saga 模式成本高。需确认产品是否接受"部分成功"的语义，还是需要严格的全部成功/全部失败。

### 问题 6 — commitNodeDirectory 原子性无法保证

- **分类**：**A（有效）**
- **理由**：与 14-2.2 和 14-4.2 相关。逐层提交中间目录的原子性问题真实存在。修复建议（基于路径 hash 的 Redis 锁）比 14 报告中的修复建议更具体。
- **跨报告重叠**：与 14-2.2（事务性缺失）、14-4.2（逐层提交版本爆炸）密切相关但不完全相同。

### 问题 7 — 缓存事件风暴风险

- **分类**：**A（有效）**
- **理由**：审查对 Pub/Sub 模式的三个风险场景分析逻辑正确。`clearRoleCache` 触发全局 `USER_PERMISSIONS` 版本更新确实可能导致缓存雪崩。修复建议（事件去重 ID、更精确的缓存失效）合理。代码中 5 秒时间窗口是缓解措施但不消除风险。

### 问题 8 — initPromise 竞态条件

- **分类**：**A（有效）**
- **理由**：与 14-2.4 完全重合。代码验证确认 `onModuleInit` 中异步设置 `initPromise` + `ensureInitialized` 可能覆盖 `initPromise` 的逻辑缺陷。回复建议的惰性初始化模式（initPromise 为 null 时重新设置、失败时清除）正确。
- **跨报告重叠**：与 14-2.4 完全重合。

### 问题 9 — 邮箱验证码 Redis 原子性

- **分类**：**A（有效）**
- **理由**：`get` + `incr` + `del` 三步非原子的问题描述准确，高并发下同一验证码可被多次使用。Lua 脚本修复建议正确且 Redis 3.0.6 支持。

### 问题 10 — FileLockService TOCTOU 竞态

- **分类**：**B（无需修复）**
- **理由**：审查承认代码在 catch 中捕获 `EEXIST` 进行重试。`existsSync` 预检查虽是多余的竞态窗口，但 `flag: 'wx'` + `EEXIST` 捕获已提供正确的原子性保证。去掉 `existsSync` 的修复建议合理但影响微小。Windows 文件时间精度问题可忽略。

### 问题 11 — 缓存击穿保护缺失

- **分类**：**A（有效）**
- **理由**：`getOrLoad` 无互斥机制在缓存过期瞬间会产生惊群效应。singleflight 修复建议是标准解决方案，成本适中。

### 问题 12 — RateLimiter 超时后任务继续执行

- **分类**：**A（有效）**
- **理由**：JavaScript Promise 不可取消导致超时后任务仍在执行的描述准确。超时后 `processNext` 递归调用可能超过 `maxConcurrent` 限制的分析逻辑正确。AbortController 修复建议合理。

### 问题 13 — 令牌黑名单 Redis 事件监听器未移除

- **分类**：**B（无需修复）**
- **理由**：NestJS 模块的 `onModuleDestroy` 在应用关闭时调用，`redis.quit()` 关闭连接后事件监听器自然失效。HMR 场景下模块重新初始化在实践中极少使用。`removeAllListeners` 是良好实践但风险极低。

### 问题 14 — Promise.race 超时定时器未清除

- **分类**：**B（无需修复）**
- **理由**：`setTimeout` 未清除导致定时器引用持续存在，但不会造成功能问题或资源泄漏（Promise 已 resolve）。jest `detectOpenHandles` 的影响属测试环境问题，可在测试配置中解决。

### 问题 15 — copyNodeRecursive 事务边界缺失

- **分类**：**A（有效）**
- **理由**：递归复制中无事务包裹，中间失败无法回滚。修复建议（`$transaction` 包裹或"先标记后清理"模式）合理。

### 问题 16 — refreshToken 无并发刷新保护

- **分类**：**B（无需修复）**
- **理由**：刷新轮转模式（rotation）是最佳实践，但当前多标签页并发刷新的影响范围极小——最坏情况是一个客户端刷新失败需重新登录。`Unique constraint` 捕获 + 前端重试机制已提供基本保护。优先级低。

### 问题 17 — RoleInheritanceService 串行缓存清除

- **分类**：**A（有效）**
- **理由**：`for...of` 串行在活跃用户数量大时性能差。`Promise.all` 并发修复建议成本低。

### 问题 18 — 配额缓存无跨实例一致性

- **分类**：**D（需用户确认）**
- **理由**：审查承认"当前的内存缓存作为 L1 缓存可接受"。是否升级到 Redis 缓存取决于部署架构和多实例一致性需求。

### 问题 19 — DirectoryAllocator 文件锁可靠性

- **分类**：**B（无需修复）**
- **理由**：审查承认 NFS 场景不可靠但对于本地文件系统可接受。当前部署假设为本地文件系统。
