# 复用优先原则

**核心规则**: 新增任何代码前，必须先搜索是否已有类似实现。这是最高优先级的编码约束。

## 前端搜索清单

新增以下类型代码时，按对应目录搜索：

| 新增类型 | 先搜索 | 搜索方式 |
|---------|--------|---------|
| UI 组件 | `src/components/ui/` → `src/components/common/` | 按功能关键词搜索（Modal、Table、Button...） |
| 工具函数 | `src/utils/` | 按函数名关键词搜索 |
| 自定义 Hook | `src/hooks/` | 按 hook 名关键词搜索（useXxx） |
| 类型定义 | `src/types/` | 检查同名/相似类型是否已存在 |
| Store | `src/stores/` | 检查已有 store 是否覆盖当前状态 |
| API 调用函数 | `src/api/` 或 `src/api-sdk/` | 检查 SDK 是否已生成对应端点 |

## 后端搜索清单

| 新增类型 | 先搜索 | 说明 |
|---------|--------|------|
| Service 方法 | 同级 Service 类 | 检查是否已有类似业务逻辑 |
| Controller 端点 | 同级 Controller | 避免重复路由 |
| DTO | `src/*/dto/` | 检查是否已有同名/相似 DTO |
| Guard/Interceptor | `src/common/` | 通用能力优先放 common |
| 工具函数 | `src/common/utils/` 或 `src/common/services/` | 搜索函数名关键词 |

## 判断标准

满足以下**任一**条件时，优先复用并改进，不新建：

1. **功能相同** — 实现完全相同的业务逻辑
2. **功能相似 > 70%** — 核心逻辑高度重叠，通过参数化/配置即可覆盖
3. **已有组件可扩展** — 添加 props 即可满足新需求
4. **已有方法可重载** — 添加可选参数即可覆盖新场景

## 何时允许新建

只有以下情况才新建：

1. **搜索无结果** — 经过完整搜索确认无已有实现
2. **功能差异 > 70%** — 核心逻辑差异过大，强行复用会导致组件失控膨胀
3. **模块边界约束** — 受制于 DDD 限界上下文，Cross-context 不宜直接复用领域对象

## 复用不完美的已有实现

遇到"已有组件/方法不够完美"时：

1. ✅ **复用 + 改进**: 在已有组件上添加 props/参数来支持新场景
2. ✅ **复用 + 重构**: 如果已有实现耦合度高，先重构使其可复用，再使用
3. ❌ **重写一套**: 不要因为"不够完美"就重新实现

## 示例

### ❌ 错误：不搜索直接新建
```
// AI 直接写了一个新的 Modal 组件
function MyCustomModal({ isOpen, onClose, children }) {
  // 完整实现...
}
// 但 src/components/ui/Modal.tsx 已经存在完全相同的功能
```

### ✅ 正确：先搜索再复用
```
// 1. 搜索发现 src/components/ui/Modal.tsx 存在
// 2. 查看其 API：<Modal open={} onClose={} title={} children={} />
// 3. 如果功能不够，添加新 props 扩展，而不是重写
```
