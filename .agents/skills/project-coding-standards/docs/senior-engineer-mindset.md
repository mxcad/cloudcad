# 高级工程师思维

AI 在编写代码时必须具备的思维模式。这些不是"建议"，是强制性行为准则。

## 核心原则

### 1. 先观察，再动手

接到任何任务后，先回答三个问题：
- 项目中**有没有**已存在的类似代码可以参照？
- 同级模块是**怎么组织**的？（目录结构、命名约定、Service 拆分方式）
- 这个改动会**影响**哪些已有模块？

**只有在回答完这三个问题后，才开始写代码。**

### 2. 遵循已有模式，不孤立设计

新增功能时，找同级模块中最相似的一个，照它的模式来：

| 要新增 | 先看 | 照搬什么 |
|--------|------|---------|
| 后端新 Controller | 已有 Controller（如 `auth.controller.ts`） | 路由版本号、装饰器风格、错误处理方式 |
| 后端新 Service | 已有 Service（如 `login.service.ts`） | 注入方式、方法签名风格、审计日志写法 |
| 前端新页面 | 已有页面组件 | 布局结构、数据获取方式、loading/error 处理 |
| 前端新 Hook | 已有 Hook（如 `useProjectPermissions.ts`） | 命名约定（`useXxx`）、参数/返回值模式 |

### 3. 简单优于复杂

- 能用 20 行解决的，不写 50 行。
- 能用已有组件组合实现的，不造新组件。
- 能让调用方传参解决的，不加抽象层。
- **每增加一层抽象，必须能用一句话说出具体好处。** 说不出 = 不必要的抽象。

### 4. 重构后代码量应减少

正常情况下，重构的目标是消除重复、简化逻辑、降低耦合——这些都意味着**代码量减少**。

- ✅ 消除了重复代码 → 行数减少
- ✅ 合并了相似逻辑 → 行数减少
- ✅ 删除了不必要的抽象 → 行数减少
- ❌ 加了更多抽象层 → 行数反而增加（通常是过度工程）

## 正确 vs 错误思维

| 场景 | ❌ AI 常犯的错误 | ✅ 高级工程师的做法 |
|------|-----------------|-------------------|
| 接到新功能 | 直接开始写代码 | 先搜索已有类似实现，理解项目模式 |
| 发现可复用代码 | "不完美，我重写一套" | 复用 + 改进（加 props/参数） |
| 重构一个模块 | 增加多层抽象，代码变多 | 消除重复，代码变少，一眼能看懂 |
| 写样式 | `color: #6366f1; background: white` | `color: var(--primary-500); background: var(--bg-primary)` |
| 写弹窗 | `zIndex: 9999` | `zIndex: Z_LAYERS.MODAL` |
| 新建文件 | 随意在某个目录创建 | 检查目录约定，放入正确位置 |
| 后端写业务逻辑 | 写在 Controller 里 | 写在 Service 里，Controller 只做路由委托 |

## 自检清单

完成任何代码变更后，在做提交前自问：

1. **有重复的代码吗？** 如果有，哪里能合并？
2. **有硬编码的值吗？** （颜色、z-index、配置项、字符串）
3. **代码量比原来多了吗？** 如果是重构，多了就是问题。
4. **有可以复用的已有组件/Service 吗？** 我搜索过了吗？
5. **一个新人能一眼看懂这段代码吗？** 看不懂就说明太复杂了。
6. **模块之间耦合低吗？** 改 A 模块需要同时改 B 吗？

如果不能坚定地回答"是"，就停下来重新思考。
