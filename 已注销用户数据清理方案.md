# 已注销用户数据清理方案

## 一、问题分析

### 当前状态

用户注销（软删除）后，以下数据残留在数据库中：

| 数据类型                              | 残留情况  | 影响                     |
| ------------------------------------- | --------- | ------------------------ |
| 项目（ownedNodes）                    | 保留      | 孤儿项目，无法管理       |
| 个人空间（personalSpaceKey = userId） | 保留      | 占用存储空间             |
| 文件节点（FileSystemNode）            | 保留      | 持续占用存储             |
| 项目成员关系（ProjectMember）         | 保留      | 污染成员列表             |
| 审计日志（AuditLog）                  | 保留      | 无实际影响，但占用数据库 |
| RefreshToken                          | ✅ 已清理 | -                        |

---

## 二、清理策略设计

### 2.1 清理时机

**延迟清理机制**：用户注销后不立即清理，保留一段"冷静期"（可配置），原因：

- 用户可能后悔，支持账户恢复
- 给管理员审核和干预的时间
- 避免误操作导致数据丢失

**建议配置**：

- `USER_CLEANUP_DELAY_DAYS`: 30 天（默认）
- 可通过环境变量覆盖

---

### 2.2 清理范围与策略

| 数据类型               | 清理策略        | 说明                         |
| ---------------------- | --------------- | ---------------------------- |
| **项目（ownedNodes）** | 软删除 → 硬删除 | 先标记删除，冷静期后物理删除 |
| **个人空间**           | 软删除 → 硬删除 | 同项目处理                   |
| **文件存储**           | 物理删除        | 释放磁盘空间                 |
| **项目成员关系**       | 物理删除        | 已注销用户不应保留在成员列表 |
| **审计日志**           | 物理删除        | 用户数据已注销，日志无意义   |
| **RefreshToken**       | ✅ 注销时已清理 | 无需处理                     |

---

### 2.3 清理流程

```
用户注销（软删除）
    ↓
设置 deletedAt = NOW()
    ↓
等待 USER_CLEANUP_DELAY_DAYS 天
    ↓
定时任务扫描过期用户（deletedAt < NOW() - delay）
    ↓
执行清理：
  1. 删除项目成员关系
  2. 软删除用户拥有的项目/空间
  3. 物理删除审计日志
  4. 标记文件存储待清理
    ↓
等待 STORAGE_CLEANUP_DELAY_DAYS 天
    ↓
StorageCleanupService 清理物理存储
```

## 三、技术实现方案

### 3.1 新增服务：`UserCleanupService`

**位置**：`packages/backend/src/common/services/user-cleanup.service.ts`

**核心方法**：

```
cleanupExpiredUsers()
  ├── 查询过期用户（deletedAt < NOW() - delay）
  ├── 对每个用户执行：
  │   ├── cleanupProjectMemberships()    // 清理项目成员关系
  │   ├── softDeleteOwnedProjects()      // 软删除拥有的项目
  │   ├── deleteAuditLogs()              // 删除审计日志
  │   └── markStorageForCleanup()        // 标记存储待清理
  └── 返回清理统计
```

**关键逻辑**：

1. **清理项目成员关系**
   - 查询用户作为成员的所有项目
   - 物理删除 `ProjectMember` 记录
   - 记录日志

2. **软删除拥有的项目**
   - 查询用户拥有的所有根节点（`ownerId = userId AND isRoot = true`）
   - 设置 `deletedAt` 和 `projectStatus = DELETED`
   - 子节点通过 Prisma 级联自动处理

3. **删除审计日志**
   - 物理删除 `userId` 关联的所有 `AuditLog` 记录

4. **标记存储待清理**
   - 查询用户拥有的所有文件节点
   - 设置 `deletedFromStorage = NOW()`
   - 由现有 `StorageCleanupService` 后续清理

---

### 3.2 新增调度器：`UserCleanupScheduler`

**位置**：`packages/backend/src/common/schedulers/user-cleanup.scheduler.ts`

**调度配置**：

- 执行频率：每天凌晨 3 点执行
- Cron 表达式：`0 3 * * *`
- 使用 `@nestjs/schedule` 的 `@Cron()` 装饰器

**执行逻辑**：

```
@Cron('0 3 * * *')
async handleCleanup()
  ├── 检查是否有正在运行的清理任务（防并发）
  ├── 调用 UserCleanupService.cleanupExpiredUsers()
  ├── 记录清理结果到日志
  └── 发送通知（可选，管理员配置）
```

---

### 3.3 配置项

**环境变量**（`.env`）：

```env
# 用户数据清理延迟天数（注销后保留天数）
USER_CLEANUP_DELAY_DAYS=30

# 是否启用定时清理任务
ENABLE_USER_CLEANUP=true

# 清理任务执行时间（Cron 表达式）
USER_CLEANUP_CRON="0 3 * * *"
```

**配置文件**（`packages/backend/src/config/configuration.ts`）：

```typescript
{
  userCleanup: {
    delayDays: process.env.USER_CLEANUP_DELAY_DAYS || 30,
    enabled: process.env.ENABLE_USER_CLEANUP !== 'false',
    cronExpression: process.env.USER_CLEANUP_CRON || '0 3 * * *',
  }
}
```

## 四、安全与容错

### 4.1 事务保护

- 每个用户的清理操作在单个事务中执行
- 失败时回滚，确保数据一致性

### 4.2 错误处理

- 单个用户清理失败不影响其他用户
- 记录详细错误日志
- 返回部分成功的统计信息

### 4.3 并发控制

- 使用 Redis 分布式锁防止并发执行
- 锁 key: `user_cleanup:lock`
- 锁超时: 1 小时

### 4.4 日志记录

- 清理前：记录待清理用户数量
- 清理中：记录每个用户的处理结果
- 清理后：汇总统计（成功/失败数量、删除数据量）

## 八、风险与注意事项

### ⚠️ 风险

| 风险               | 影响 | 缓解措施                   |
| ------------------ | ---- | -------------------------- |
| 误删数据           | 高   | 30 天冷静期，事务保护      |
| 清理失败           | 中   | 错误隔离，日志记录         |
| 性能影响           | 中   | 分批处理，避免大事务       |
| 存储空间未及时释放 | 低   | 依赖 StorageCleanupService |

### 📝 注意事项

1. **首次执行**：可能有大量历史注销用户需要清理，建议手动触发并监控
2. **数据库备份**：首次执行前建议备份数据库
3. **存储清理**：文件物理删除依赖 `StorageCleanupService` 的延迟配置
4. **审计合规**：某些场景可能要求保留审计日志，需根据业务调整
