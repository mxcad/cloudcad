# 通用代码审查提示词

## 使用方法

将此提示词发送给 AI，然后附上你想审查的代码文件路径或代码片段。

---

## 提示词

```
请对以下代码进行严格审查。审查时请遵循以下原则：

## 审查原则

1. **只报告真正的 bug** - 不要报告代码风格、命名偏好、"可以优化"等非问题
2. **验证问题存在** - 每个问题必须对应代码中的具体位置，能指出文件名和行号
3. **评估实际影响** - 说明这个 bug 会导致什么具体后果
4. **给出修复建议** - 提供可直接应用的代码修复

## 审查维度

### 1. 功能性 Bug
- 参数是否正确传递
- 返回值是否正确处理
- 分支逻辑是否有遗漏
- 边界条件是否处理

### 2. 数据一致性
- 数据库操作是否有事务保护
- 缓存与数据库是否一致
- 状态更新是否原子性

### 3. 安全漏洞
- 用户输入是否验证
- SQL 注入 / XSS 风险
- 权限检查是否完整
- 敏感信息是否泄露

### 4. 错误处理
- 异常是否被捕获
- 错误信息是否对用户友好
- 失败后是否有清理操作

## 输出格式

对于发现的每个问题，请使用以下格式：

### 问题 N: [问题标题]

**位置**: 文件名:行号

**代码**:
```typescript
// 问题代码
```

**问题**: 具体描述什么错了

**影响**: 会导致什么后果

**修复**:
```typescript
// 修复后的代码
```

## 要求

- 如果没有发现问题，直接说"未发现 bug"，不要凑数
- 不要报告"建议优化"、"代码可以更简洁"等非必要问题
- 每个问题必须是真正会导致功能异常的 bug
```
